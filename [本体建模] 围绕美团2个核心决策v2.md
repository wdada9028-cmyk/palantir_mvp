# 本体建模 v2

## 1. 文档目标

本文档在 `v1` 基础上补充英文实体名、属性名对应的中文释义，便于评审、沟通和后续落库。

本版本体优先支撑两个核心决策：

1. 在开始进场安装前，如何安排到货，不导致货物积压
2. 到货后，如何规划安装到哪个楼、哪一层、哪个机房、哪个机柜落位

同时，当前版本是 MVP 版本体，因此属性设计遵循一个额外原则：

- 只保留直接支撑关键决策、排期计算和约束判断的必要属性
- 对展示类、命名类、补充说明类属性，先不作为 MVP 必填属性
- 后续如果进入产品化或主数据治理阶段，再逐步补齐

---

## 2. 当前建模边界

1. `PoD`：核心业务对象
2. 暂时不展开 PoD 内部机柜，只在未来存在按柜活动时再下钻
3. 一个 `PoD` 对应一个 `PoDPosition`
4. `PoDPosition` 的表达方式采用类似 `A机房-01位`
5. 里程碑分成两类：
   - `RoomMilestone`：机房里程碑
   - `FloorMilestone`：楼层里程碑
6. `RoomMilestone` 的完成标准是：该机房要求数量的 `PoD` 在里程碑时间前完成 `移交`
7. `FloorMilestone` 的完成标准是：该楼层下若干机房全部完成移交
8. `移交` 是明确的二级活动节点，可作为倒排锚点
9. 一级活动用于分类分组，真正实例化与排期的是二级活动
10. 活动排期分两层：
    - `PoD` 内部活动排期
    - 施工队执行分配排期
11. 一个 `PoD` 到货后即可独立安装，第一版不区分到货时间和可安装时间
12. 施工队能力采用固定规则，不做复杂资源优化
13. 第一版不照搬原始设备颗粒度到货模型，而是以 `PoD` 为业务抽象粒度

## 4. Object Types

说明：
- 以下“关键属性”均按 MVP 口径保留
- 如果某属性只用于展示、标识美化或解释增强，而不影响当前两个核心决策，则暂不纳入 MVP 必要属性

## 4.1 项目与目标层

### `Project`

中文释义：项目。表示一个面向客户的交付项目，是所有对象的业务聚合根。

关键属性：
- `project_id`：项目ID
- `project_name`：项目名称
- `project_status`：项目状态
- `product_type`：产品形态
- `cooling_mode`：冷却方式
- `project_scene`：项目场景
- `planned_start_date`：计划开始日期

### `RoomMilestone`

中文释义：机房里程碑。表示机房级里程碑约束。

语义定义：
- 某个机房，在某个时间前，完成指定数量 `PoD` 的 `移交`

关键属性：
- `room_milestone_id`：机房里程碑ID
- `project_id`：所属项目ID
- `room_id`：目标机房ID
- `proposed_by`：提出方
- `target_pod_count`：目标PoD数量
- `completion_event_code`：完成事件编码
- `completion_event_name`：完成事件名称
- `due_time`：截止时间
- `priority`：优先级
- `status`：状态

说明：
- `completion_event_name` 第一版固定为 `移交`

### `FloorMilestone`

中文释义：楼层里程碑。表示楼层级里程碑约束。

语义定义：
- 某个楼层下的若干机房，在某个时间前全部完成移交

关键属性：
- `floor_milestone_id`：楼层里程碑ID
- `project_id`：所属项目ID
- `floor_id`：目标楼层ID
- `proposed_by`：提出方
- `due_time`：截止时间
- `required_room_count`：要求完成的机房数量
- `completed_room_count`：已完成的机房数量
- `status`：状态

## 4.3 设备与物流层

### `PoD`

中文释义：PoD交付单元。表示交付、到货、安装、移交的核心业务对象。

关键属性：
- `pod_id`：PoD ID
- `project_id`：所属项目ID
- `pod_code`：PoD编码
- `pod_type`：PoD类型
- `pod_status`：PoD状态
- `building_id`：所在大楼ID
- `floor_id`：所在楼层ID
- `room_id`：所在机房ID
- `position_id`：所在落位ID
- `planned_arrival_time`：计划到货时间
- `actual_arrival_time`：实际到货时间
- `planned_install_start_time`：计划安装开始时间
- `actual_install_start_time`：实际安装开始时间
- `planned_handover_time`：计划移交时间
- `actual_handover_time`：实际移交时间

状态建议：
- `InTransit`：在途
- `ArrivedWaitingInstall`：已到货待安装
- `Installing`：安装中
- `Installed`：已安装
- `HandedOver`：已移交

### `Shipment`

中文释义：发货单。表示一次 PoD 发货安排。

关键属性：
- `shipment_id`：发货ID
- `project_id`：所属项目ID
- `shipment_no`：发货单号
- `pod_id`：对应PoD ID
- `planned_ship_time`：计划发货时间
- `planned_arrival_time`：计划到货时间
- `actual_arrival_time`：实际到货时间
- `shipment_status`：发货状态

### `ArrivalEvent`

中文释义：到货事件。表示一次 PoD 到货事实。

关键属性：
- `arrival_event_id`：到货事件ID
- `pod_id`：对应PoD ID
- `arrival_time`：到货时间
- `arrival_status`：到货状态
- `receiving_location`：收货地点
- `confirmed_by`：确认人

### `ArrivalPlan`

中文释义：到货建议方案。表示系统对到货时间的建议方案。

关键属性：
- `arrival_plan_id`：到货方案ID
- `project_id`：所属项目ID
- `pod_id`：对应PoD ID
- `recommended_arrival_time`：建议到货时间
- `latest_safe_arrival_time`：最晚安全到货时间
- `earliest_useful_arrival_time`：最早有意义到货时间
- `backlog_risk_level`：积压风险等级
- `plan_status`：方案状态

## 4.4 活动与排期层

### `ActivityTemplate`

中文释义：活动模板。表示标准二级活动模板。

关键属性：
- `template_id`：活动模板ID
- `l1_code`：一级活动编码
- `l1_name`：一级活动名称

- `completion_flag`：是否为完成节点标记

说明：

### `ActivityDependencyTemplate`

## 4.7 MVP 暂不强制的补充属性

以下属性对理解和展示有帮助，但不是当前 MVP 支撑两个核心决策所必需，因此先不作为必填属性：

- `customer_name`：客户名称
- `planned_handover_date`：计划移交日期
- `milestone_name`：里程碑名称
- `building_code`：大楼编码
- `floor_name`：楼层名称
- `room_name`：机房名称
- `room_code`：机房编码
- `position_name`：落位名称
- `crew_name`：施工队名称
- `reason_summary`：原因摘要

补充说明：
- `position_code` 目前保留，因为像 `A机房-01位` 这样的编码本身就承载了位置决策语义
- `building_name` 保留，是为了让“到哪个楼”的建议可直接读懂
- `project_name` 保留，是为了跨项目查看时仍有基本可识别性

## 5. Link Types

### 5.1 项目与空间关系

- `Project HAS Building`：项目包含大楼
- `Building HAS Floor`：大楼包含楼层
- `Floor HAS Room`：楼层包含机房
- `Room HAS PoDPosition`：机房包含PoD落位

### 5.2 项目与里程碑关系

- `Project HAS RoomMilestone`：项目包含机房里程碑
- `Project HAS FloorMilestone`：项目包含楼层里程碑
- `RoomMilestone CONSTRAINS Room`：机房里程碑约束机房
- `FloorMilestone CONSTRAINS Floor`：楼层里程碑约束楼层
- `FloorMilestone AGGREGATES RoomMilestone`：楼层里程碑聚合机房里程碑

### 5.3 项目与 PoD 关系

- `Project DELIVERS PoD`：项目交付PoD
- `PoD ASSIGNED_TO Building`：PoD分配到大楼
- `PoD ASSIGNED_TO Floor`：PoD分配到楼层
- `PoD ASSIGNED_TO Room`：PoD分配到机房
- `PoD ASSIGNED_TO PoDPosition`：PoD分配到落位

### 5.4 PoD 与物流关系

- `Project HAS Shipment`：项目包含发货单
- `Shipment SHIPS PoD`：发货单运输PoD
- `PoD HAS ArrivalEvent`：PoD具有到货事件
- `ArrivalPlan APPLIES_TO PoD`：到货方案作用于PoD

### 5.5 PoD 与活动关系

- `PoD HAS ActivityInstance`：PoD具有活动实例
- `ActivityTemplate GENERATES ActivityInstance`：活动模板生成活动实例
- `ActivityInstance USES SLAStandard`：活动实例使用标准SLA
- `ActivityInstance DEPENDS_ON ActivityInstance`：活动实例依赖活动实例
- `ActivityDependencyTemplate DEFINES ActivityInstance`：活动依赖模板定义活动实例关系
- `PoDSchedule CONTAINS ActivityInstance`：PoD排期包含活动实例
- `PoDSchedule APPLIES_TO PoD`：PoD排期作用于PoD

### 5.6 施工执行关系

- `Project HAS Crew`：项目包含施工队
- `Crew EXECUTES WorkAssignment`：施工队执行施工分配
- `WorkAssignment ASSIGNS PoD`：施工分配指向PoD
- `WorkAssignment OCCURS_IN Room`：施工分配发生在机房
- `WorkAssignment OCCURS_AT PoDPosition`：施工分配发生在落位

### 5.7 决策与解释关系

- `PlacementPlan APPLIES_TO PoD`：落位方案作用于PoD
- `PlacementPlan REFERENCES PoDPosition`：落位方案引用落位
- `ConstraintViolation REFERENCES PoD`：约束冲突关联PoD
- `ConstraintViolation REFERENCES RoomMilestone`：约束冲突关联机房里程碑
- `ConstraintViolation REFERENCES FloorMilestone`：约束冲突关联楼层里程碑
- `DecisionRecommendation REFERENCES ArrivalPlan`：决策建议引用到货方案
- `DecisionRecommendation REFERENCES PlacementPlan`：决策建议引用落位方案
- `DecisionRecommendation REFERENCES ConstraintViolation`：决策建议引用约束冲突

## 7. 一句话总结

第一版本体的核心思想是：

以 `PoD` 作为核心业务对象，以 `RoomMilestone / FloorMilestone` 作为外部约束，以 `移交` 作为倒排锚点，以 `PoDPosition` 作为落位对象，以 `ActivityInstance + WorkAssignment` 分别承载工序逻辑和施工执行逻辑，最终支撑“到货建议”和“落位建议”两类关键决策，并能够解释风险与不可执行原因。
